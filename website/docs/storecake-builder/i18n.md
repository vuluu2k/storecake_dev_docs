# Internationalization (i18n)

`builderx_spa` uses **vue-i18n 9** with the **i18n-ally** VS Code extension to manage translation keys.

## Layout

```text
src/i18n/
├── index.js                # createI18n, autoload locales
├── locales/
│   ├── vi.json
│   ├── en.json
│   ├── th.json
│   └── …
└── helpers.js              # Format helpers (if any)
```

- Default locale: `vi`.
- Fallback: `en`.
- Source language for i18n-ally: `vi`.

## i18n-ally settings

Already pinned in `.vscode/settings.json`:

```json
"i18n-ally.localesPaths": ["src/i18n/locales"],
"i18n-ally.keystyle": "nested",
"i18n-ally.sourceLanguage": "vi",
"i18n-ally.sortKeys": true,
"i18n-ally.translate.engines": ["google"],
"i18n-ally.regex.usageMatchAppend": [
  "t\\.success\\(\\s*['\"]({key})['\"]",
  "t\\.error\\(\\s*['\"]({key})['\"]"
]
```

The regex `usageMatchAppend` also picks up `t.success('...')` and `t.error('...')` helpers.

## Usage

### Options API

```vue
<template>
  <a-button>{{ $t('product.create_new') }}</a-button>
</template>

<script>
export default {
  methods: {
    onCreate() {
      this.$t('product.create_new');
      this.$message.success(this.$t('product.created'));
    },
  },
};
</script>
```

### Composition API

```vue
<script setup>
import { useI18n } from 'vue-i18n';

const { t, locale } = useI18n();
const label = t('product.create_new');
</script>
```

### From plain modules

```js
import i18n from '@/i18n';
i18n.global.t('product.create_new');
```

## Key conventions

- Nest by feature: `product.create_new`, `order.status.completed`,…
- Do not reuse the same key for different contexts.
- Keys are snake_case English; translations vary by locale.
- When deleting a key, remove it from **every** locale file (no stale keys).
- `vi` is the source — i18n-ally fills other locales automatically.

## Lint

`eslint-plugin-i18n-json` covers `src/i18n/locales/*.json` via `lint-staged`:

```json
"lint-staged": {
  "src/i18n/locales/*.json": "eslint --fix",
  "*.{js,ts,vue}": "eslint --fix"
}
```

The plugin keeps key order in sync, warns on mismatched `\{placeholder\}` between locales, and flags missing keys against `sourceLanguage`.

## Adding a new locale

1. Create `src/i18n/locales/<code>.json` (copy from `vi.json`).
2. Use i18n-ally → "Translate missing keys".
3. Register the locale in `src/i18n/index.js` (`messages` map) and in the locale-picker UI (`useLocaleStore`).
4. Mirror the locale list in the backend if storefront translations are involved.

## Pitfalls

- Avoid dynamic keys built from interpolated variables (for example, calling `$t` with a template string that includes a variable) — i18n-ally cannot detect them, so they get pruned by cleanup. Prefer mapping a closed set of keys instead:

  ```js
  const KEY_BY_TYPE = {
    info: 'product.type.info',
    sale: 'product.type.sale',
  };
  $t(KEY_BY_TYPE[type]);
  ```
- For numbers and dates, use `$n` and `$d` with `numberFormats` / `datetimeFormats` configured in `src/i18n/index.js`.
- For lazily-loaded locales, call `i18n.global.setLocaleMessage(locale, msgs)` before `setLocale(locale)`.
